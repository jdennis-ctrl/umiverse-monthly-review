Feature Focus: Alpha Environment - What We Built & Why It Matters

GWI's engineering & data teams were sharing the same test environment, causing disruptions, slowdowns and conflicts. We led the work to split them into two separate, purpose-built environments, so each team can work without impacting the other.

Phase 1 - Analysis & Planning

→Investigated what was needed to make Alpha a fully working environment for engineers
→Identified all third-party tools (Stripe, Salesforce, Segment, Mixpanel) that needed separate Alpha configurations
→Mapped out all 13 UM services that needed to be deployed and tested in Alpha

Phase 2 - Environment Setup

→Deployed all 13 UM services (users, teams, billing, sign-in and more) into the new Alpha environment
→Set up dedicated Stripe payment sandboxes so billing features could be safely tested without touching real data
→Created dedicated analytics tracking tokens for Alpha so test activity would not pollute production reports
→Updated deployment pipelines so Alpha automatically gets the latest code whenever production is updated
→Set up error monitoring (Sentry) and a dedicated Slack alerts channel for Alpha
→Configured Google and Microsoft sign-in (SSO) to work correctly in Alpha

Phase 3 - Security & Authentication

→Fixed a critical bug where engineers were automatically logged in as the wrong person when switching between Alpha and Staging
→Gave Alpha its own unique login tokens so it no longer shares session data with Staging
→Updated all 8 front-end applications to use the new Alpha-specific tokens during a careful transition period
→Cleaned up the old legacy tokens once all services had safely switched over
→Fixed a sign-out bug where users were being redirected to the Staging login page instead of Alpha
→Resolved a sign-in failure that prevented normal (non-SSO) logins from working in Alpha

Phase 4 - Data Clean-up

→Alpha was mirroring production users, causing real customers to receive "Goodbye" account deletion emails - we identified and fixed this
→Built automated clean-up jobs for 8 services to ensure Alpha only contains appropriate test data
→Ran clean-up scripts across all Alpha databases to remove production data that had been incorrectly mirrored
→Ensured free user account settings were preserved correctly after each database clean-up cycle
→Removed a safeguard that was blocking account clean-up emails in Alpha
→Ensured Mixpanel analytics in Alpha no longer shows inactivity events that could be confused with production activity

Phase 5 - Go Live & Validation

→Updated and ran the full automated test suite against Alpha to confirm everything works as expected
→Created and validated all test user accounts needed for ongoing QA work in Alpha
→Fixed performance issues where the admin user list was loading slowly in Alpha compared to Staging
→Verified Stripe billing features work correctly end-to-end in Alpha
→Confirmed deployment pipelines are working - production deployments now correctly trigger Staging updates
→Alpha is now the primary engineering test environment, freeing Staging for data team use only

Feature Focus: Alpha Environment - Further Breakdown

Feature Focus: What Did We Actually Deliver?

A full breakdown of the six work streams delivered to make Alpha a fully functional, secure and independent engineering environment. Click any item to read more.

🔧

Environment Configuration

All 13 UM services deployed & configured in Alpha
+
Services included: users, teams, organisations, billing, sign-in, admin, devices, permissions, bundles, registrations, service tokens, SSO and analytics - all deployed and verified working in Alpha.
Deployment pipelines updated to sync Alpha with production
+
Previously Alpha and Staging were kept in sync. We have now moved to Phase 2 where Staging receives deployments triggered by Production, making Alpha a true independent environment.
All deployment variables checked and corrected across all services
+
Every service's configuration settings were reviewed and corrected to ensure they pointed to the right Alpha resources rather than Staging or Production ones.
💳

Payments & Third Parties

Dedicated Stripe payment sandboxes created for Alpha
+
Two Stripe sandboxes were created (one for GWI, one for Trendstream) with all required products, pricing and webhooks configured so billing can be safely tested without affecting real payments.
Analytics tools (Segment & Mixpanel) given dedicated Alpha tokens
+
Alpha now has its own Segment and Mixpanel tokens, meaning test activity in Alpha no longer appears in production analytics dashboards, keeping reporting clean and accurate.
Error monitoring (Sentry) configured with dedicated Slack alerts
+
Sentry is now set up to detect errors in Alpha and send alerts to a dedicated #alerts-internal-alpha Slack channel, so the team can respond to issues quickly without noise from other environments.
🧹

Data Clean-up & Safety

Fixed: real customers receiving "Goodbye" emails from Alpha
+
Alpha was mirroring production users. When Alpha's clean-up process ran, it treated those mirrored users as inactive and sent them real "account deletion" emails. We identified the root cause and resolved it so this can never happen again.
Automated data clean-up jobs built for 8 services
+
We built automated jobs for users, teams, organisations, billing, devices, permissions, SSO and bulk actions - each one safely removes production data from Alpha after each data sync, keeping the environment clean.
Alpha databases cleaned and verified
+
All Alpha databases were cleaned of production user data, with QA test organisations and users preserved. The calculations team was notified to sync their local database copy after each clean-up.
🔒

Security & Authentication

Fixed: engineers auto-logged in as wrong user in Alpha
+
Alpha and Staging were sharing the same login cookie name. When an engineer used both environments, their sessions crossed over, causing them to appear as a different user. We gave Alpha its own unique cookie name to eliminate this entirely.
All 8 front-end apps updated to use Alpha-specific login tokens
+
Every front-end application (sign-in, admin, account settings, navigation and more) was updated in a careful phased rollout, supporting both old and new tokens temporarily before the legacy ones were safely removed.
Google & Microsoft SSO configured and working in Alpha
+
Single Sign-On via Google and Microsoft now works correctly in Alpha with dedicated OAuth credentials, so engineers can test SSO login flows without impacting the production sign-in experience.
🧪

Testing & Quality Assurance

Full automated test suite updated to run against Alpha
+
All automated tests across every UM repository were updated to target Alpha instead of Staging. This means every code change is now tested against the correct environment before it reaches production.
All QA test user accounts created & validated in Alpha
+
Every test user account required for automated testing was created, configured and validated in Alpha, including SSO users which required additional setup via the admin tool.
Performance issue fixed - admin user list loading slowly in Alpha
+
QA discovered that the admin user list was taking significantly longer to load in Alpha than Staging, causing test timeouts. This was investigated and resolved so tests run reliably.
🐛

Bugs Found & Fixed

Sign-out redirected to wrong environment
+
When signing out of Alpha admin, users were being taken to the Staging login page instead of Alpha. This was confusing for engineers and has been corrected.
Normal (non-SSO) login not working in Alpha
+
Standard email and password login was failing in Alpha - the system was signing users out immediately after a successful login. This was traced to a cookie conflict and resolved.
Stripe pricing table not displaying in Alpha account settings
+
The Stripe pricing and plan selection page was not rendering correctly in Alpha. The configuration was pointing to the wrong environment settings and has been corrected.

Feature Focus: Change URL for Contact Redirect within Admin

When users in Admin hit a permissions barrier, they are shown a 'contact support' link. This link was pointing to a Zendesk page that no longer exists, leaving users stranded with no way to get help. We updated every instance of this broken link across all admin pages to redirect users to the correct internal GWI support page instead, which provides them with the process to contact Product Experience.

The Problem & The Fix

⚠️ Before

Users hitting a permissions wall were shown a 'contact support' link pointing to a Zendesk page that was no longer active. Clicking it led to a broken page, leaving users with no route to get help or take action.

zendesk.com/... (broken & inactive)
→

✅ After

The link now redirects users to the GWI internal Product Experience page - a live, working destination. All references to Zendesk have been removed and replaced with the correct destination across every affected page.

intranet.gwiu.gwi.com - Product Experience
What We Updated & How It Was Verified
⚙️

What We Updated

Updated the broken contact support hyperlink to the new GWI internal page across all React, Elm and Ruby admin pages
Removed all references to Zendesk from page copy wherever they appeared across all three technology stacks
Changes were applied consistently so the user experience is the same regardless of which admin page they are on
🎯

What QA Checked on Each Page

Triggering a permissions-restricted action surfaces the contact support link correctly
Clicking the link redirects correctly to the GWI internal Product Experience page
No remaining references to Zendesk visible anywhere on the page
